The specification of binary integer values is a common task usually accomplished using hexidecimal literals. This is an unnecessary and error-prone indirection between what is intended and what is accomplished.

Runtime Computation

The simplest method of converting binary literals to values is to perform the conversion at runtime. 

#include <stdio.h>
#include <stdlib.h>

unsigned long binary(const char* x) {
    unsigned long r = 0;
    while (*x) (r *= 2) += *x++ - '0';
    return r;
}

int main(void) {
    printf("%lu\n", binary("101010"));
    return EXIT_SUCCESS;
}

Besides the inefficiency of runtime computation, the major drawback of this approach is that the result cannot be used as a compile-time constant.

(On the other hand, the major advantage of this approach is that it can do the computation at runtime--which is important in its own right. 

int main(int argc, char* argv[]) {
    if (argc < 2) {
        return EXIT_FAILURE;
    }
    printf("%lu", binary(argv[1]));
    return EXIT_SUCCESS;
}

Neither template metaprogramming nor preprocessor metaprogramming can achieve that.)

Template Metaprogramming

In C++, template metaprogramming can be used to specify binary literals by reinterpreting the value of decimal literal (containing nothing but 0's and 1's). 

#include <iostream>

template<class T, T V> struct map_integral {
    static const T value = V;
};

template<class T, T V> const T map_integral<T, V>::value;

template<unsigned long x> struct binary
    : map_integral<unsigned long, binary<x / 10>::value << 1 | x % 10> { };

template<> struct binary<0> : map_integral<unsigned long, 0> { };

int main(void) {
    std::cout << binary<11111111>::value << '\n'; // prints 255
    return 0;
}

However, there are a few drawbacks to using the template mechanism for this purpose, the most obvious of which is that this approach can only be used in C++. The other drawbacks are more subtle....

Integer literals cannot contain enough digits to specify the full range of the greatest-width integer. If, for example, an unsigned long is a 32-bit integer, a decimal literal would require thirty-two digits to fully specify each bit--which is well outside the range of a 32-bit integer. 

binary<10101010101010101010101010101010>::value // error!

Thus, in order to specify such a value, the bits must be separated into groups and processed semi-independently. The following implementation uses four groups of at most eight bits (note that, for simplicity, the value of a particular bit group isn't checked for overflow): 

#include <iostream>

template<class T, T V> struct map_integral {
    static const T value = V;
};

template<class T, T V> const T map_integral<T, V>::value;
    
template<int A, int B = -1, int C = -1, int D = -1> struct binary
    : map_integral<
        unsigned long,
        binary<A, B, C>::value << 8 | binary<D>::value
    > { };

template<int A, int B, int C> struct binary<A, B, C>
    : map_integral<
        unsigned long,
        binary<A, B>::value << 8 | binary<C>::value
    > { };

template<int A, int B> struct binary<A, B>
    : map_integral<
        unsigned long,
        binary<A>::value << 8 | binary<B>::value
    > { };

template<int A> struct binary<A>
    : map_integral<
        unsigned long,
        binary<A / 10>::value << 1 | A % 10
    > { };

template<> struct binary<0> : map_integral<unsigned long, 0> { };

int main(void) {
    std::cout
        << binary<11111111>::value                               << '\n'
        << binary<11111111, 11111111>::value                     << '\n'
        << binary<11111111, 11111111, 11111111, 11111111>::value << '\n';
    return 0;
}

(With a little help from preprocessor metaprogramming, the boilerplate of this metafunction implementation can be extended to handle (nearly) arbitrary numbers of bit groups. However, doing so would probably be overkill, as the greatest-width integer type is unlikely to grow fast (or differ from platform to platform) enough to justify the boilerplate of the preprocessor generation itself.)

The reliance on the greatest-width integer type illustrates another drawback of the template-based approach. The compiler cannot perform arbitrary-precision arithmetic (at least, not directly). As such, there is a (relatively low) limit on the maximum integer width on which the compiler can perform arithmetic. This isn't terrible--most results need to be stored in some integer variable anyway. Nevertheless, the limitation exists.

The most subtle drawback of the template-based approach is that the value of each bit group must be either 0 or begin with 1. Otherwise, the literal is treated as octal instead of decimal--which fouls up the entire mechanism: 

binary<00000010>::value // 8 (not 2)

Thus, binary literals cannot be specified (as they typically are) with leading 0's to fill up some specific width. That limitation is not only annoying, but is fundamentally error-prone because there is no way to verify the original forms of the input values. (One possible solution to this problem is to alter the implementation to expect hexidecimal numbers instead of decimal numbers. This would allow leading 0's at the inconvenience of prefixing each value with 0x or 0X.)

Though the implementation can be specified in a number of different ways, these drawbacks are fundamental. The second, third, and fourth drawbacks are all the result of a single root issue. The above code acts on integral values instead of acting on the literals themselves.

Enter the Preprocessor...

As it happens, there is a language mechanism that can operate (to a degree) on literal values--macros.

Given some data structure of bits (i.e. 0's and 1's), it is relatively easy to calculate a decimal value by accumulating the bits. 

unsigned long binary(const char* x) {
    unsigned long r = 0;
    while (*x) {
        (r *= 2) += *x++ - '0';
    }
    return r;
}

Each time a bit is found, the algorithm doubles the running result and adds the value of the bit. Instead of accumulating the elements of a data structure by manually looping (or recursing), a generalized notion of "accumulation" can be used--std::accumulate. 

#include <cstring>
#include <numeric>

unsigned long binary(const char* x) {
    struct local {
        static unsigned long exec(unsigned long x, char bit) {
            return x * 2 + (bit - '0');
        }
    };
    return std::accumulate(x, x + std::strlen(x), 0UL, &local::exec);
}

This is a higher-order transformation which yields a design that has greater similarity to a typical preprocessor-based design than the original iterative algorithm. Note that the above transformation is actually two transformations. The first is the abstraction of looping itself (e.g. the loop body is executed repeatedly until some supplied predicate yields false). The second is the abstraction of the terminating condition--when the input sequence is exhausted.

Direct iteration, such as the while loop above, is impossible with the preprocessor. However, the code can be rewritten recursively--which can be mirrored with the preprocessor. 

unsigned long binary(const char* x, unsigned long r = 0) {
    return *x ? binary(x + 1, r * 2 + (*x - '0')) : r;
}

It is possible to implement a preprocessor-based version at any of these (not counting the iterative version) levels of abstraction. (The following examples all use a string data structure to specify bits.)

Direct translation of the recursive version requires a much more detailed understanding of how recursion is generalized in Chaos. For the sake of comparison, it is listed here: 

#include <chaos/preprocessor/arbitrary/add.h>
#include <chaos/preprocessor/arbitrary/demote.h>
#include <chaos/preprocessor/arbitrary/mul.h>
#include <chaos/preprocessor/control/iif.h>
#include <chaos/preprocessor/recursion/basic.h>
#include <chaos/preprocessor/recursion/expr.h>
#include <chaos/preprocessor/string/core.h>

#define BINARY(string) BINARY_S(CHAOS_PP_STATE(), string)
#define BINARY_S(s, string) \
    CHAOS_PP_EXPR_S(s)(BINARY_I( \
        CHAOS_PP_OBSTRUCT(), CHAOS_PP_NEXT(s), string, (0) \
    )) \
    /**/
#define BINARY_INDIRECT() BINARY_I
#define BINARY_I(_, s, string, x) \
    CHAOS_PP_IIF _(CHAOS_PP_STRING_IS_CONS(string))( \
        CHAOS_PP_EXPR_S _(s)(BINARY_INDIRECT _()( \
            CHAOS_PP_OBSTRUCT _(), CHAOS_PP_NEXT(s), \
            CHAOS_PP_STRING_TAIL _(string), \
            CHAOS_PP_ARBITRARY_ADD _( \
                CHAOS_PP_ARBITRARY_MUL _(x, (2)), \
                (CHAOS_PP_STRING_HEAD _(string)) \
            ) \
        )), \
        CHAOS_PP_ARBITRARY_DEMOTE _(x) \
    ) \
    /**/

BINARY(1 0 1 0) // 10

This version is the lowest-level, and, as such, it is likely the most efficient. However, the code is dominated by code to emulate recursion rather than code to calculate the value of a binary literal.

A cleaner implementation can be designed if the level of abstraction is increased. The next step up from low-level recursion is the use of a higher-order looping construct, such as CHAOS_PP_WHILE, which eliminates the need to deal with recursion. 

#include <chaos/preprocessor/arbitrary/add.h>
#include <chaos/preprocessor/arbitrary/demote.h>
#include <chaos/preprocessor/arbitrary/mul.h>
#include <chaos/preprocessor/array/elem.h>
#include <chaos/preprocessor/control/while.h>
#include <chaos/preprocessor/facilities/as_args.h>
#include <chaos/preprocessor/recursion/expr.h>
#include <chaos/preprocessor/string/core.h>

#define BINARY(string) BINARY_S(CHAOS_PP_STATE(), string)
#define BINARY_S(s, string) \
    CHAOS_PP_ARBITRARY_DEMOTE(CHAOS_PP_ARRAY_ELEM( \
        1, \
        CHAOS_PP_EXPR_S(s)(CHAOS_PP_WHILE_S( \
            s, \
            PRED CHAOS_PP_AS_ARGS(2), \
            OP CHAOS_PP_AS_ARGS(2), \
            (2, (string, (0))) \
        )) \
    )) \
    /**/

#define PRED(s, string, x) CHAOS_PP_STRING_IS_CONS(string)
#define OP(s, string, x) \
    (2, ( \
        CHAOS_PP_STRING_TAIL(string), \
        CHAOS_PP_ARBITRARY_ADD( \
            CHAOS_PP_ARBITRARY_MUL(x, (2)), \
            (CHAOS_PP_STRING_HEAD(string)) \
        ) \
    )) \
    /**/

BINARY(1 0 1 0) // 10

This version, as far as clarity is concerned, is much better than the raw recursive version. However, there are still a few annoyances. The state on which the loop operates includes both the string of bits and the running result. This means that the operation has to operate on both in order for the predicate to eventually yield 0. Likewise, when the loop does terminate, the result is both the nil string and the arbitrary-precision value that holds the result. Thus, the actual result must be extracted (in this case, with CHAOS_PP_ARRAY_ELEM).

All of these annoyances can be eliminated if the level of abstraction is increased yet again. The next step up is to use a higher-order algorithm that generalizes the notion of accumulation--which is also known as "folding." In this example, the algorithm of choice is CHAOS_PP_FOLD_LEFT. 

#include <chaos/preprocessor/algorithm/fold_left.h>
#include <chaos/preprocessor/arbitrary/add.h>
#include <chaos/preprocessor/arbitrary/demote.h>
#include <chaos/preprocessor/arbitrary/mul.h>
#include <chaos/preprocessor/recursion/expr.h>
#include <chaos/preprocessor/string/core.h>

#define BINARY(string) BINARY_S(CHAOS_PP_STATE(), string)
#define BINARY_S(s, string) \
    CHAOS_PP_ARBITRARY_DEMOTE( \
        CHAOS_PP_EXPR_S(s)(CHAOS_PP_FOLD_LEFT_S( \
            s, OP, (CHAOS_PP_STRING) string, (0) \
        )) \
    ) \
    /**/

#define OP(s, bit, x) \
    CHAOS_PP_ARBITRARY_ADD( \
        CHAOS_PP_ARBITRARY_MUL(x, (2)), (bit) \
    ) \
    /**/

BINARY(1 0 1 0) // 10

The above example eliminates the problems associated with the template-based approach. It unifies the syntax by eliminating the need for groups of bits, it minimizes the "maximum value" problem, and it avoids the "octal literal" issue. However, this implementation has its own drawback--the binary literal itself must be encoded in a string data structure (e.g. 1 0 1 0 as opposed to 1010). That isn't too terrible, but it isn't ideal.

An Alternative Approach

Maybe the idea of having groups of bits isn't such a bad idea after all. It is possible to support groups of bits with the preprocessor as well. Such a syntax would look something like 

BINARY(10101010 10101010)

The major drawback of this approach is that it requires 256 macros (for eight-bit groups) to transform a group into something that can be processed. Beyond that, the main problem is that 10101010 10101010 (and similar) is not a valid string because each element has more than one character. In fact, it isn't any library-defined data structure. Thus, the special form must be manually converted (using a set of 256 macros) before the result can be accumulated. This is accomplished with a CHAOS_PP_WHILE loop that transforms each "element" into a string. 

#include <chaos/preprocessor/algorithm/fold_left.h>
#include <chaos/preprocessor/arbitrary/add.h>
#include <chaos/preprocessor/arbitrary/demote.h>
#include <chaos/preprocessor/arbitrary/mul.h>
#include <chaos/preprocessor/config.h>
#include <chaos/preprocessor/control/while.h>
#include <chaos/preprocessor/detection/is_nullary.h>
#include <chaos/preprocessor/facilities/split.h>
#include <chaos/preprocessor/logical/compl.h>
#include <chaos/preprocessor/recursion/expr.h>
#include <chaos/preprocessor/string/core.h>

#if CHAOS_PP_VARIADICS

#define BINARY(x) BINARY_S(CHAOS_PP_STATE(), x)
#define BINARY_S(s, x) \
    CHAOS_PP_ARBITRARY_DEMOTE( \
        CHAOS_PP_EXPR_S(s)(CHAOS_PP_FOLD_LEFT_S( \
            s, ACC, \
            (CHAOS_PP_STRING) CHAOS_PP_SPLIT( \
                0, \
                CHAOS_PP_EXPR_S(s)(CHAOS_PP_WHILE_S( \
                    s, PRED, OP,, x \
                )) \
            ), \
            (0) \
        )) \
    ) \
    /**/

#define ACC(s, bit, x) \
    CHAOS_PP_ARBITRARY_ADD( \
        CHAOS_PP_ARBITRARY_MUL(x, (2)), (bit) \
    ) \
    /**/

#define PRED(s, r, x) \
    CHAOS_PP_COMPL(CHAOS_PP_IS_NULLARY(x())) \
    /**/

#define OP(s, r, x) \
    r CHAOS_PP_SPLIT(0, BYTE_ ## x), CHAOS_PP_SPLIT(1, BYTE_ ## x) \
    /**/

#define BYTE_00000000 0 0 0 0 0 0 0 0,
#define BYTE_00000001 0 0 0 0 0 0 0 1,
#define BYTE_00000010 0 0 0 0 0 0 1 0,
// ...
#define BYTE_11111111 1 1 1 1 1 1 1 1,

BINARY(00000101) // 5
BINARY(10001000) // 136
BINARY(11111111) // 255
BINARY(11111111 00000000) // 65280

#endif

(The BYTE_ boilerplate can be pre-generated with a small program such as the following: 

#include <stdio.h>
#include <stdlib.h>

void byte(int depth) {
    static char s[9] = { 0 };
    if (depth == 8) {
        printf(
            "#define BYTE_%s %c %c %c %c %c %c %c %c,\n", s,
            s[0], s[1], s[2], s[3], s[4], s[5], s[6], s[7]
        );
    }
    else {
        s[depth] = '0';
        byte(depth + 1);
        s[depth] = '1';
        byte(depth + 1);
    }
    return;
}

int main(void) {
    byte(0);
    return EXIT_SUCCESS;
}

This implementation has a couple of properties that are good and a couple that are bad--depending on perspective. It forces each element to be a full eight bits--which is a primitive form of validation. The main concern with the above is that it wastes time doing the conversion from the special form to a string. For a small number of groups, this overhead isn't too bad, but perhaps it can be eliminated altogether....

The central point of the following implementation is that a separate special form conversion can be avoided if a new data type is defined--in which case it is no longer a special form. The following code first implements a data structure XS (which stands for "extensible string") and then implements BINARY on top of that. The XS data structure is similar to a string except that elements can be identifiers or numbers that are greater than a single character provided an appropriate "expander" is defined. For example, 

#define EXPANDER_abc (a b c)
#define EXPANDER_xyz (x y z)

XS_HEAD(abc xyz) // a

CHAOS_PP_CAST(
    CHAOS_PP_STRING, (XS) abc xyz \
)
// (CHAOS_PP_STRING) a b c x y z

This data structure is extensible in the sense that user-defined "expanders" can be easily written. For example, 

#define EXPANDER_Paul (P a u l)
#define EXPANDER_Mensonides (M e n s o n i d e s)

#define OP(s, x) CHAOS_PP_TO_UPPER(x)

CHAOS_PP_EXPR(CHAOS_PP_TRANSFORM(
    OP, (XS) Paul _ Mensonides
))
// (XS) P A U L _ M E N S O N I D E S

Given such a data structure definition (and the appropriate EXPANDER_xxx definitions), input such as 10101010 10101010 can be processed directly. The potential downside is that the primitive validation has been lost because the XS data type is perhaps too extensible for this purpose. 

#include <chaos/preprocessor/algorithm/fold_left.h>
#include <chaos/preprocessor/arbitrary/add.h>
#include <chaos/preprocessor/arbitrary/demote.h>
#include <chaos/preprocessor/arbitrary/mul.h>
#include <chaos/preprocessor/cat.h>
#include <chaos/preprocessor/control/iif.h>
#include <chaos/preprocessor/detection/is_nullary.h>
#include <chaos/preprocessor/detection/is_unary.h>
#include <chaos/preprocessor/logical/compl.h>
#include <chaos/preprocessor/recursion/expr.h>
#include <chaos/preprocessor/string/core.h>
#include <chaos/preprocessor/tuple/rem.h>

#define XS_TAG(tag) tag

#define XS_IS_NIL(xs) CHAOS_PP_IS_NULLARY(xs())

Summary

Without language support, there is no perfect solution to binary literals. This article provides a few different ways that they can be specified, each with various drawbacks, but no one solution is ideal. Luckily, the purpose of the article isn't to solve that problem completely. Rather, the problem is merely an example that can be used to illustrate preprocessor metaprogramming methodology and to exercise Chaos. Binary literals, though convenient, aren't really all that useful. However, if one knows how to use the tools and techniques available, there is very little that cannot be done.

Postscript

The astute reader might notice that there is nothing particularly special about binary literals (as opposed to literals of other bases). The runtime model might look something like the following: 

#include <ctype.h>
#include <stdio.h>
#include <stdlib.h>

unsigned long helper(char c) {
    if (isdigit(c)) {
        return c - '0';
    }
    else switch (tolower(c)) {
        case 'a': return 10;
        case 'b': return 11;
        case 'c': return 12;
        case 'd': return 13;
        case 'e': return 14;
        case 'f': return 15;
        default:
            return exit(EXIT_FAILURE), 0;
    }
}

unsigned long convert(const char* x, int base) {
    if (base > 16) exit(EXIT_FAILURE);
    unsigned long r = 0;
    while (*x) (r *= base) += helper(*x++);
    return r;
}

int main(void) {
    printf(
        "%lu\n%lu\n%lu\n%lu\n",
        convert("ff", 16),
        convert("123", 10),
        convert("00000010", 2),
        convert("00000010", 8)
    );
    return EXIT_SUCCESS;
}
